## Environment / integrity notes

- All source repos were read **read-only** at the pinned SHAs in
  [`_sources.md`](./_sources.md). `$REPOS/linux` (`e43ffb69`, v7.1-rc6) and
  `$REPOS/llvm-project` (`73802c2e`) were pre-existing detached checkouts reused
  without fetching or modification; `c/valgrind`, `cpp/drmemory`, and
  `cpp/googletest` were cloned this session (absent per the recon inventory).
- **SHA verification (2026-07-11).** Every pinned SHA was re-checked with
  `git -C <path> rev-parse HEAD` and matches the worker reports. Annotated-tag
  caveat: `git rev-parse VALGRIND_3_26_0` returns the tag **object** (`bc7e4193`);
  the pinned **commit** is `218cee2f` (= the checked-out HEAD), which is what the
  ledger records. LDC is read at tag `v1.41.0` via `git show`, but the working
  checkout is `feat/wasm` @ `f4d2f831` (`git describe` = `v1.42.0-91-gf4d2f831c3`)
  — **do not switch that branch** (other work depends on it).
- **Toolchain version skew (disclose per page).** On the primary box the IR
  instrumentation is LDC's **LLVM 18.1.8** pass, but the **linked runtime is GCC
  15.2's libsanitizer** (a _newer_ compiler-rt merge — a strict API superset of
  compiler-rt 18 for the `__asan_/__lsan_/__sanitizer_/__tsan_` prefixes, W1/W2),
  while the **source read is compiler-rt HEAD `73802c2e`** (post-18, carries the
  new `adaptive_delay`/RTSan/TySan surfaces GCC 15.2 lacks). Every page mirrors
  the CPU-PMU habit of a `> [!WARNING]` disclosing this three-way skew.
- **Privilege posture.** All x86_64-linux experiments ran unprivileged;
  mac-bsn ran unprivileged with SIP enabled; nothing was escalated or disabled.
  The one Windows-adjacent hardware fact (Apple-clang ASan on M4) is the recon
  smoke test; no Windows claim is hw-verified.
