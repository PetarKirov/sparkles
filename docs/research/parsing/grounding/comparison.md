## Discrepancies

**⚠ C1 — chumsky zero-copy mislabeled "1.0" (comparison.md:33 and :116).**
The matrix cell (`grep -n "✅ (1.0)" comparison.md` → **line 33**) and the §3 Zero-copy
bullet ("`[chumsky][chumsky] 1.0`", **line 116**) attribute chumsky's zero-copy capability
to version **1.0**. Per the chumsky deep-dive ledger (`rust-chumsky.md` rows 8,9):
zero-copy parsing "carried since **0.10.0** (2025-03-22)" and the current released version
is **0.13.0**; **1.0 is unreleased** (only `1.0.0-alpha.0` exists — the page itself states
this correctly at line 160, "0.x + 1.0-alpha"). Labeling the feature "(1.0)" implies it is
gated behind an unreleased version when it has shipped in every release since 0.10.
**Fix:** change "✅ (1.0)" → "✅ (0.10+)" at line 33 and "chumsky 1.0" → "chumsky 0.10+"
(or "chumsky's rewrite line") at line 116. Severity: minor (defensible as "the 1.0-rewrite
lineage", but internally inconsistent with line 160's own "1.0-alpha" framing).

**⚠ C2 — "Parsec (… GHC …)" adopter is dubious (comparison.md:158).**
The Decades-deep ecosystem row lists Parsec as load-bearing in "Cabal, GHC, Dhall"
(`grep -n "Cabal, GHC, Dhall" comparison.md` → **line 158**). Cabal does use Parsec; but
**GHC's own parser is Happy-generated** (alex/happy), not Parsec, and Dhall uses
_megaparsec_, not parsec proper. There is **no `haskell-parsec` ledger** yet
(`grounding/` has no parsec file; index.md marks it "pending"), so this adopter list is
not backed by a completed grounding pass. **Fix:** verify the Parsec adopter roll-call
when the haskell-parsec deep-dive is grounded; drop or qualify "GHC" (consider "Cabal,
Idris, Dhall-family") — this is a web-adopter framing, low severity, but the GHC claim is
likely wrong. (Tracked as a wave-1 gap, not a hard error introduced by this page.)
