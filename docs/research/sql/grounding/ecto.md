## Discrepancies

- **#D1 (cross-page, effect-model framing) — `ecto.md` metadata + Effect-model section
  vs `concepts.md`.** The concepts "landscape at a glance" table
  (`sql/concepts.md:259`) groups Ecto under the **"Effect-system SQL"** family with
  effect/async model **"effect value (typed errors)"**. The Ecto page deliberately
  corrects this at the level of _mechanism_: Ecto's `Repo` calls are **eager and
  blocking** and return **tagged tuples** (`{:ok, _}` / `{:error, _}`), with adapter
  failures raised as exceptions — there is **no** `IO`/`ZIO`/`Effect`/`ConnectionIO`
  value and **no** type-level error channel (grounded at `lib/ecto/repo.ex:1864-1866`,
  `:145-148`, `:2382-2387`; no effect type in `lib/`). The concepts table's grouping is
  a _family/positioning_ convenience (Ecto shares the "explicit persistence, no ORM
  identity-map" destination), whereas the page's per-library characterization is the
  accurate effect model. Not a factual error in either place; flagged because the page
  reads as a mild correction of the shared table. The scope NOTE and Weaknesses make the
  distinction explicit.

- **#D2 (example composition) — `ecto.md` Multi example.** The page pipes
  `Multi.new() |> … |> Repo.transact()` as one pipeline; the source defines
  `PasswordManager.reset/2` returning the `Multi` (`lib/ecto/multi.ex:58-67`) and then
  runs it with `Repo.transact(PasswordManager.reset(account, params))`
  (`:71`). Semantically identical; the page inlines the two for brevity. No factual
  impact.
